**[ НАЗВАНИЕ ПРОЕКТА ]**

“Платформа для подготовки к олимпиадам по информационной безопасности”

**[ ПРОБЛЕМНОЕ ПОЛЕ ]**

В сети крайне мало платформ для изучения безопасности веб-приложений на практике. 
У существующих платформ есть ряд серьезных недостатков: мало заданий, однотипные задания на эксплуатацию CVE (подобные задания редко встречаются на олимпиадах и в реальной жизни. Польза от их решения минимальная), старые, неактуальные задания, 90% заданий доступно только по подписке (из России без зарубежной карты оплатить подобные подписки нельзя).

В результате у целевой аудитории отсутствует удобный и доступный способ регулярно тренировать практические навыки веб-безопасности.

**[ ЗАКАЗЧИК / ПОТЕНЦИАЛЬНАЯ АУДИТОРИЯ ]**

Студенты и школьники, которые хотят участвовать в олимпиадах по информационной безопасности, или начать заниматься багхантингом и тестированием веб-приложений.

**[ АППАРАТНЫЕ / ПРОГРАММНЫЕ ТРЕБОВАНИЯ ]** 

Продукт представляет собой веб-сайт и не требует установки.  Для работы необходим
доступ в интернет и современный браузер.

**[ ФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ ]**

Программный продукт предоставляет следующие возможности:

* Доступ к обучающим курсам по безопасности веб-приложений, структурированным по уровням сложности;

* Доступ к практическим заданиям по веб-безопасности;

* Скачивание заданий в виде архивов с файлами для развёртывания уязвимых веб-приложений;

* Проверка правильности найденного флага через интерфейс сайта;

* Навигация по курсам и заданиям с фильтрацией по уровню сложности и темам;

* Публикация новых заданий и обновление существующих материалов;

**[ ПОХОЖИЕ / АНАЛОГИЧНЫЕ ПРОДУКТЫ ]**

Анализ 3 программных продуктов, которые максимально приближены к заданному функционалу, показал, что:

* TryHackMe - мало заданий на веб, много платных заданий, однотипные задания.
* PicoCTF - мало заданий, устаревшие, однотипные задания, слабая теория.
* HTB - однотипные, скучные задания, большинство функций платные.

**[ ИНСТРУМЕНТЫ РАЗРАБОТКИ, ИНФОРМАЦИЯ О БД ]**

*	C# ASP.NET - для бэкенда
*	HTML / CSS / JS / - для фронтенда
*	PostgreSQL - бд
*   Nginx - api gateway
*   Docker - оркестрация, контейнеризация

**[ ЭТАПЫ РАЗРАБОТКИ ]**

* Проектирование. Перед реализацией нужно определить архитектуру платформы и стек (1-2 дня)
* Частичная реализация архитектуры. Первичная настройка уже готовых сервисов по типу баз данных, api gateway и т.д. Подготовка репозитория к деплою, настройка сервера, подготовка сервера к деплою (1-2 дня)
* Реализация бэкенд части проекта. Нужно написать серверную часть кода платформы, связать её с подготовленными сервисами, настроить для неё деплой (1-2 недели)
* Реализация тестов. Нужно написать нагрузочные и unit тесты, связать их с пайплайном деплоя (3-4 дня)
* Реализация фронтенд части проекта. Нужно написать клиентскую часть кода платформы, связать её с серверной частью, настроить деплой (1-2 недели)
* Доработка написанного кода. Исправление багов, оптимизация кода, реализация доп. функционала по типу сбора метрик и т.д. (1-2 недели)
* Наполнение контентом. Нужно придумать и написать много заданий и курсов для платформы, качественно подобрать теоретический материал (3-4 месяца)

**[ ВОЗМОЖНЫЕ РИСКИ ]**

* Нехватка времени из-за большого объёма работ, особенно на этапе наполнения контентом. Чтобы решить проблемы с нехваткой времени нужно чёткое планирование этапов, приоритизация задач, постепенное расширение функционала + от некотырых идей в случае чего можно отказаться.
* Низкое качество или устаревание учебных материалов. Нужно использовать доверенные источники, проверять репутацию авторов, консультироваться с более опытными людьми в сфере ИБ.
* Низкая заинтересованность целевой аудитории. Нужно получить обратную связь от ЦА (например лицеисты, которые занимаются олимпиадами по ИБ) на этапе добавления первых заданий и курсов.
